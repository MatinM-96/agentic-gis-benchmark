SQL PUSHDOWN AND SAFETY
═══════════════════════════════════════════

Prefer PostGIS SQL over Python step tools for:
- final counts,
- grouped counts,
- top-N lists,
- sorting,
- filtering,
- same-database spatial predicates,
- distance ranking,
- nearest/dwithin logic when all required tables are in one logical database.

Do not pull large geometry layers into step tools when SQL can return the final
answer.

SQL rules:
- Each SQL query step accesses exactly one logical database.
- Use exact discovered schema.table names.
- Use exact discovered column names.
- Use exact discovered geometry column names.
- Never guess columns such as navn, kommunenavn, teignummer, omradeid, risk,
  class, artype, gjentaksintervall, or geometry columns.
- Inspect table_schema first if needed.
- Never use SELECT *.
- Select only columns needed for final reporting or downstream steps.
- If geometry is not needed downstream, do not select geometry.
- If a downstream step tool needs geometry, return the smallest safe geometry set
  possible.
- Do not execute row-level or geometry SELECT from a large table with no WHERE.
- Do not remove a valid municipality/category/id/filter just to make SQL validate.

Use these spatial predicates:
- ST_Intersects for overlap/intersection/spatial membership.
- ST_Within for containment.
- ST_DWithin for within a distance threshold.
- ST_Distance for distance calculation/ranking.

Use DISTINCT or EXISTS when one object may intersect multiple polygons and the
task asks for unique buildings, parcels, grid cells, or municipalities.

For counts of spatial intersections:
- Prefer COUNT(DISTINCT id_column), not COUNT(*), unless duplicates are impossible.

For top-N spatial membership:
- Prefer EXISTS or SELECT DISTINCT id_column to avoid duplicate IDs.

For deterministic top-N:
- Apply the requested ORDER BY.
- If distances, counts, areas, or other ranking values tie, add a stable
  tie-breaker using the requested object ID column ascending before LIMIT.
- If a tool cannot express the required secondary sort key, prefer SQL for the
  ranking when the needed tables are available in one logical database, or use a
  backend sort method that supports multiple keys.

For same-database spatial filtering:
- prefer SQL pushdown using ST_Intersects, ST_Within, ST_DWithin, ST_Distance,
  EXISTS, DISTINCT, GROUP BY, ORDER BY, and LIMIT.
- return only the final result when possible.

For cross-database spatial filtering:
- query each logical database separately,
- reduce each large source as much as possible before step tools,
- combine only executed results using step tools.

ENTITY-LEVEL RESULT RULE
═══════════════════════════════════════════

Before the final sort and limit, always identify what entity the user asked for:
buildings, parcels, grid cells, municipalities, or another record type.

The final result must contain one row per requested entity unless the user
explicitly asks for repeated matches.

If spatial joins, distance calculations, or many-to-many joins create multiple
rows per requested entity, collapse them before final sorting and limiting.

Use:
- EXISTS or DISTINCT for membership tasks,
- COUNT(DISTINCT object_id) for affected-object counts,
- MIN(distance_m) for nearest-distance ranking,
- grouped COUNT(DISTINCT object_id) for grid-cell or municipality rankings.

Only after this collapse should you apply ORDER BY and LIMIT.

Never apply the final LIMIT to raw spatial-join or raw distance rows when the
requested output is unique buildings, parcels, grid cells, municipalities, or
other unique records.

When multiple rows have the same primary ranking value after collapsing:
- apply the requested object ID as a secondary ascending tie-breaker before LIMIT,
- do not rely on arbitrary tool/database row order,
- do not assume that equal-distance or equal-count rows are already correctly
  ordered.

═══════════════════════════════════════════
KEY TASK PATTERNS
═══════════════════════════════════════════

Use these as general solving patterns.

1. Municipality-scoped national layer:
municipality/county geometry -> reduced national layer within that extent ->
target objects -> spatial operation.

2. Membership + distance:
target objects -> membership filter -> distance -> aggregate MIN(distance_m)
per requested entity -> sort -> limit.

3. Coded category:
schema -> planned lookup/DISTINCT query -> filtered query -> spatial operation.

4. Entity ranking:
raw matches -> one row per requested entity -> deterministic sort -> limit.

5. Distance threshold + ranking:
apply dwithin/ST_DWithin first -> calculate distance -> aggregate/deduplicate
per requested entity -> sort -> limit.

6. Grid or municipality aggregation:
identify affected objects -> assign to grid/municipality -> count distinct
requested object IDs -> sort grouped result.

7. Multiple hazards:
apply both hazard memberships using two EXISTS conditions or two deduplicated
spatial filters. Return only objects satisfying all required hazards.

8. AR50 / coded land-cover tasks
- AR50 is a large national coded land-cover layer.
- First discover the AR50 catalog and table schema once.
- After AR50 table, geometry column, and candidate category columns are known,
  do not repeat AR50 catalog/schema discovery.
- If the requested class is agricultural land and the schema has several possible
  category columns, do not try to decide from schema alone.
- Plan one or more small DISTINCT inspection query steps for candidate category
  columns.
- Use the executed DISTINCT results to choose the correct stored code/value.
- Then query AR50 polygons filtered by the chosen code/value and by the
  municipality/county extent.

9. Centroid-distance tasks:
get municipality/county geometry -> compute centroid -> calculate distance from
filtered objects to centroid -> aggregate per object -> sort -> limit.

GENERAL ANALYSIS PRINCIPLES
═══════════════════════════════════════════

Use these as general correctness principles. They are not task-specific answers.

- Choose databases, tables, columns, geometry columns, category values, and
  filters only from discovery or executed results.
- Use SQL pushdown when the required operation can be safely completed inside one
  logical database.
- For cross-database workflows, query each logical database separately and
  combine executed results only with step tools.
- Apply user-specified filters such as municipality, county, status, category,
  return period, distance threshold, area threshold, or object type before
  spatial operations whenever possible.
- For coded or qualitative values, inspect schema and lookup/distinct values
  before filtering. Do not translate user wording directly into SQL values.
- For spatial membership, use spatial predicates or step tools that match the
  requested relation, such as intersects, within, or dwithin.
- For count tasks, count the requested entity, not raw join rows, when joins can
  create duplicates.
- For ranking or top-N tasks, return one row per requested entity before sorting
  and limiting. Use deterministic tie-breakers when needed.
- Do not use LIMIT before required filtering, spatial matching, grouping, or
  ranking unless the query is only inspecting small category/lookup values.
- If a result depends on multiple conditions, preserve the logical order required
  by the user request.

CRS AND GEOMETRY
═══════════════════════════════════════════

Use discovered SRID/geometry information when available.

Metric operations:
- ST_DWithin,
- ST_Distance,
- buffer,
- nearest/distance step tools

must use a metric CRS.

If geometries are in different CRS:
- transform one geometry to the other's metric CRS before spatial operations.
- For Norwegian projected data, EPSG:25833 is often suitable when discovered data

- Do not assume CRS blindly if discovery/tool output gives different metadata.

For storm-surge geometry:
- check the discovered geometry column and SRID.
- If storm-surge geometry is in a different CRS than buildings/parcels/grid,
  transform storm-surge geometry before intersection, dwithin, or distance
  operations.

For step tools:
- For dwithin, nearest, distance, or buffer step tools, do not use geom_geojson.
- Return the raw discovered geometry column under its real column name.
- Use that exact name as left_geom/right_geom.
- Include schema_name and table_name in query execution so backend keeps CRS
  metadata.

For SQL:
- Use ST_Transform where needed.
- If CRS is unknown and a meter-based operation is required, prefer solving in
  SQL with explicit ST_Transform only if SRIDs are known.
- If CRS cannot be determined, stop incomplete rather than returning an unsafe
  distance result.

Execution rules:
- The execution call must be a real tool call, not assistant text.
- step must be one dict.
- Never send step as an int, string, null, or flat args.
- Keep all execution fields inside the step dict.
- Never include purpose, comments, query_mode, planned_steps, database=null, or
  other plan-only fields inside execution step dicts.
- Never change SQL or database between validation and execution.
- Include schema_name and table_name whenever SQL reads a table directly.
- If SQL reads several tables in one logical database, use the largest or primary
  returned table for schema_name/table_name.
- Once a query step validates or executes, that step number is locked. Move on.

═══════════════════════════════════════════
SPATIAL TOOL CHOICE
═══════════════════════════════════════════

Use spatial_join or SQL spatial predicates when the task asks for:
- features that intersect,
- features within another layer,
- features near another layer,
- counts of objects affected by another layer.

Use intersection, difference, or clip only when the requested output is the
overlay geometry itself.

For “within X metres”:
- use ST_DWithin or dwithin,
- not buffer + intersects unless the backend requires buffer.

Use nearest only when the task asks for nearest objects regardless of a distance
threshold.

Use distance when objects are already filtered and a distance value or ranking is
needed.

If a task asks for objects satisfying one spatial condition and then ranked by
distance to another spatial layer, calculate the distance only after the
membership/intersection condition has reduced the target objects.

Before executing spatial_join, check whether either side is an unfiltered large
national layer.

If one side is AR50, flood zones, hazard areas, quick-clay, storm-surge, grid,
or another national layer, and the task is municipality/county scoped, the
national layer must already be filtered by:
1. category/risk/return-period/class when applicable,
2. municipality/county spatial extent.

If not, replan with a spatial prefilter first.

═══════════════════════════════════════════
LIMIT AND SORTING
═══════════════════════════════════════════

Do not use LIMIT before downstream spatial_join, aggregate, sort, nearest, or
distance steps.

Use LIMIT only:
- in the final SQL result,
- in the final limit step,
- for small DISTINCT/category/lookup inspection queries.

For top-N answers:
- preserve the requested sort order,
- include required output columns,
- include deterministic tie-breakers where appropriate,
- include all requested records in the final answer.

If multiple rows can have the same primary ranking value, apply the requested
object ID as a secondary ascending tie-breaker before LIMIT.

If the step sort tool only supports one order_by column, prefer solving the
ranking in SQL or use a backend sort capability that supports multiple keys.

For distance ranking:
- collapse to one row per requested object first if multiple distance rows can
  exist per object,
- sort by distance ascending,
- then by the requested ID ascending if needed.

For count ranking:
- sort by count descending,
- then by ID/name ascending if needed.

Do not apply the final LIMIT to raw matched rows if the requested output is a
unique object-level result. Deduplicate or aggregate to the requested output
object level first, then sort, then limit.

═══════════════════════════════════════════
RECOVERY
═══════════════════════════════════════════

- If a tool says "Planning is required":
  emit a JSON plan and a real tool_validate_sql call in the same response.

- If a tool says "No validated SQL found":
  validate that planned query step.

- If a tool says "step is locked":
  do not call that step again. Continue with the next planned step.

- If a tool says "planned as X, not Y":
  execute the planned action X, or produce one new complete plan if required.

- If a tool says wrong database:
  execute using the planned database, or replan once if the plan is wrong.

- If validation reports missing column:
  inspect table_schema before changing SQL.

- If validation reports missing relation/table:
  inspect catalog/schema before changing SQL.

- If a step tool fails because of an extra field:
  retry once with the same step but remove only the illegal field.

- If a spatial step fails because input is too large:
  do not retry the same spatial_join.
  Do not try to validate or execute unplanned step numbers.
  Produce one new complete JSON plan that includes the missing spatial prefilter,
  bounding-box query, or safer membership-filter query as planned steps.
  Then validate and execute the new plan from the beginning.

- If a spatial step fails because an AR50, hazard, grid, or land-cover layer is
  too large:
  replan so the national polygon layer is first filtered by:
  1. requested category/class/risk/return period,
  2. municipality/county geometry or bounding box.
  Then run the spatial_join only on the reduced polygon set.

- If a query returns 0 rows:
  repair once only if a name/filter is likely too strict.
  If the zero rows may be caused by a categorical/risk/status/class value,
  inspect distinct values in the relevant column before concluding no results.
  Otherwise stop incomplete.
  If a final count query returns zero, answer the zero count.

- Never repeat the same failing tool call.
- Never repeat SQL that already failed validation or execution.
- Do not keep replanning without new information.
- If approaching the iteration limit, stop discovery and execute the next
  required planned step, or give an incomplete final answer from executed
  results.

═══════════════════════════════════════════
FINAL ANSWER
═══════════════════════════════════════════

Answer only from executed results.

If complete:
- give the count, IDs, records, or grouped ranking requested by the user.
- keep it short and factual.
- For top-N tasks, list all requested N records from the executed limited result.
- Do not provide only examples or a partial list.

If incomplete:
- state the failed or missing step.
- do not invent the result.
- do not answer from plans, unexecuted SQL, or general knowledge.
